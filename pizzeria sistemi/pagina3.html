Comunicazione nel Web con HTTP

In questa lezione esploreremo il modello Client/Server, la struttura degli URI e il funzionamento profondo del protocollo HTTP/1.1.

🌐

Il Modello Client/Server

Il Web si basa su un'interazione tra due entità distinte:

Client (Elemento Attivo): I Web Browser (Chrome, Firefox) che utilizzano l'URL per identificare risorse e inviano richieste HTTP.
Server (Elemento Passivo): I Web Server (Apache, Nginx) che rimangono in ascolto sulla porta 80/443 e forniscono le risorse richieste.

Identificare le Risorse: URI, URL e URN

Ogni risorsa nel Web (file, immagine, query) è identificata in modo univoco.

URI (Identifier)

Il set generico di nomi o indirizzi (URL + URN).

URL (Locator)

Specifica come raggiungere la risorsa (protocollo + host + path).

URN (Name)

Nome persistente anche se la risorsa cambia posizione.

ComponenteSignificatoObbligatorio
ProtocolloHTTP, HTTPS, FTP, ecc.Sì
HostNome simbolico o IP del server.Sì
PortaDefault 80 per HTTP, 443 per HTTPS.No
Path/FilePercorso gerarchico nel file system.Sì
QueryInformazioni addizionali (es. ?id=10).No
FragmentPunto preciso nella risorsa (es. #fine).No

Caratteristiche del Protocollo

HTTP (HyperText Transfer Protocol) è un protocollo di livello Application che opera sopra il TCP/IP.

  • Stateless: Non mantiene memoria dello stato tra diverse richieste.
  • Versione 1.0: Ogni risorsa richiede una nuova connessione TCP (lento).
  • Versione 1.1: Introduce la Connessione Permanente (Keep-Alive) e il Pipelining.

HTTP Pipelining

Il client invia molteplici richieste senza attendere la risposta immediata, ottimizzando i tempi di latenza della rete.

Struttura del Messaggio HTTP

Sia la Richiesta che la Risposta condividono una struttura comune:

  1. Riga Iniziale (Request Line o Status Line)
  2. Header Lines (Metadati del messaggio)
  3. Riga Vuota (CRLF - Fondamentale per separare i metadati dai dati)
  4. Message Body (I dati effettivi, opzionale)
GET /informatica/home.html HTTP/1.1
Host: www.istitutovolta.edu
User-Agent: Mozilla/5.0
Accept: text/html

<!-- Riga vuota CRLF -->

Metodi e Risposte

Il client specifica l'azione tramite un Metodo:

  • GET Richiede una risorsa (dati nell'URL).
  • POST Invia dati al server (dati nel Body).
  • HEAD Richiede solo gli Header (per test).

La Status Line (Response)

Il server risponde con un codice numerico:

200 OK: Successo.
404 Not Found: Risorsa non trovata.
500 Error: Errore del server side.

Header e Message Body (JSON)

Gli Header definiscono come il messaggio deve essere interpretato. Ad esempio, Content-Type: application/json indica che il corpo del messaggio contiene dati strutturati in formato JSON, standard de-facto per le moderne Web API.

HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8

{
  "ID": 215,
  "Username": "denny",
  "Città": "Genova"
}
