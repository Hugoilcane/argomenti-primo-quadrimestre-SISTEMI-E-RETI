Il Livello delle Applicazioni

Analisi estesa delle applicazioni di rete. Scopriremo non solo i concetti di Client-Server, P2P e WWW, ma approfondiremo le logiche di Load Balancing, i metodi HTTP, i meccanismi TCP/UDP e l'evoluzione dei protocolli di trasferimento sicuri.

Cos'è il Livello Applicazione?

Nel modello ISO/OSI e TCP, il livello applicazione implementa i servizi di rete usati dall'utente finale. A differenza dei livelli inferiori, qui la flessibilità è massima, consentendo lo sviluppo continuo di nuovi protocolli (come HTTP, DNS, SMTP, FTP). Approfondimento: Nel modello TCP/IP, questo livello accorpa i tre livelli superiori del modello OSI (Applicazione, Presentazione e Sessione).

Processi e Socket

Un'applicazione di rete è un insieme di programmi eseguiti su più host. Per far comunicare due processi, si utilizza il Socket, formato da Indirizzo IP + Numero di Porta.

La classificazione delle porte (IANA):
• Well-Known (0-1023): Riservate a servizi standard. Es: HTTP (80), SMTP (25), DNS (53), HTTPS (443).
• Registered (1024-49151): Registrate per applicazioni specifiche (es. database).
• Dynamic/Private (49152-65535): Usate temporaneamente dal sistema operativo per i processi client in uscita (multiplexing).

Architetture di Rete

1. Client/Server Avanzato

Prevede un server sempre attivo (IP statico) che risponde ai client. I client non comunicano mai direttamente tra loro.

Gestione della Congestione: Server Farm e Load Balancing
Per evitare colli di bottiglia, colossi come Google o Amazon usano server farm (un hostname associato a più IP trasparenti al client). Questo avviene tramite Load Balancer (es. Nginx o HAProxy) che intercettano il traffico e lo smistano in base al carico (algoritmi Round-Robin o Least-Connections) ai vari server interni.
2. Peer-to-Peer (P2P) Evoluto

I nodi (peer) dialogano direttamente basandosi sul principio "dare e ricevere" (condivisione di banda e file). Può essere decentralizzato (ogni nodo è "servent"), centralizzato (server indice come Napster) o ibrido (super-peer).

Evoluzioni Moderne del P2P:
• BitTorrent: Usa un tracker per trovare i peer. Il file viene diviso in piccoli "chunk". Un client scarica frammenti da nodi diversi contemporaneamente (swarming) ed è obbligato a caricare (tit-for-tat) per poter scaricare velocemente.
• Blockchain: Un'architettura P2P completamente decentralizzata dove i peer condividono un registro crittografico immutabile anziché semplici file.

Servizi dello Strato di Trasporto: TCP vs UDP

Le applicazioni scelgono il protocollo in base ai requisiti di Affidabilità, Throughput (Banda), Temporizzazione e Sicurezza.

TCP (Transmission Control Protocol)

Utilizzato quando non è ammessa alcuna perdita di dati (FTP, HTTP, Email).

  • Three-Way Handshake: Prima di inviare dati, client e server stabiliscono la connessione scambiandosi i pacchetti SYN, SYN-ACK e ACK.
  • Controllo di congestione: Regola la velocità di trasmissione per non saturare la rete.
  • Riordino: Assicura che i pacchetti arrivino nell'ordine corretto.

UDP (User Datagram Protocol)

Protocollo senza connessione ("fire and forget"), ideale quando la velocità conta più della precisione (Streaming Audio/Video, Gaming).

  • Leggerezza: Nessun handshake iniziale, inizia a trasmettere immediatamente.
  • Nessuna garanzia: I pacchetti possono arrivare disordinati o perdersi. Non esiste controllo di congestione.
  • Intestazione minima: Solo 8 byte rispetto ai 20 byte del TCP, riducendo l'overhead.

L'Architettura del WWW

Il Web è un ipertesto multimediale distribuito basato su tre pilastri: Identificazione (URI/URL), Interazione (Protocolli) e Formato (HTML/Content-Type).

Il Cuore del Web: HTTP

L'HTTP gestisce l'interazione tra browser e server web. È basato su comandi specifici (Metodi):

  • GET: Richiede una risorsa (es. caricare una pagina).
  • POST: Invia dati al server (es. l'invio di un modulo o di una password).
  • PUT/PATCH: Aggiorna una risorsa esistente sul server.
  • DELETE: Elimina una risorsa.
Protocollo Stateless: L'HTTP "non ha memoria". Il server non ricorda le richieste precedenti di un client. Per ovviare a ciò e mantenere l'utente "loggato", sono stati inventati i Cookie e le Sessioni.

Il Protocollo FTP e l'Evoluzione della Sicurezza

Il File Transfer Protocol (RFC 959) è un'architettura client/server pura che utilizza due connessioni TCP parallele.

I Due Canali FTP

1. Connessione di Controllo (Porta 21): Per comandi, autenticazione e risposte (es. comando `RETR` o `QUIT`).
2. Connessione Dati (Porta 20): Usata unicamente per spostare i file reali.

Problema NAT/Firewall: Nell'FTP "Attivo", il server cerca di connettersi al client sulla porta dati. Poiché i router moderni bloccano il traffico in entrata, si usa oggi la modalità FTP Passivo (PASV), in cui è il client ad aprire entrambe le connessioni.

Dall'FTPS all'SFTP

L'FTP in chiaro permette facilmente lo sniffing di password. È nato l'FTPS che aggiunge un layer crittografico SSL/TLS (RFC 4217).

Il nuovo standard (SFTP):
Oggi l'approccio più usato a livello aziendale non è FTPS, ma SFTP (SSH File Transfer Protocol). Non è un'estensione dell'FTP, ma un protocollo interamente nuovo che viaggia sulla singola porta 22 del protocollo SSH. Trasferisce dati e comandi in un unico canale, cifrato e fireweall-friendly.
